개발자로서 취준하면서 기초 CS 지식들을 묻는 기술 면접도 소홀히 할 수 없다. 학교에서 전공하면서 CS로 나름 공부했었고 정처기 준비하면서 공부를 했었지만 워낙 단기기억특화라서 입력하고 한번 출력하면 잊어버린다...이런 1회성 입출력구조... 개발자로서 앞으로 다시 열심히 찾아보고 준비해보려 한다. > 틀린 답변이 있을지도 모르니 믿지 말고 다시 검토를 꼭... ### JVM 메모리 구조 자바 프로그램은 `자바프로그램 > JVM > OS > 하드웨어` 순으로 실행되는데 프로그램이 실행되면 JVM은 OS로 부터 프로그램이 필요로 하는 메모리를 할당받고 JVM은 이 메모리를 용도에 따라 여러 영역으로 나누어 관리한다. 크게 Garbage Collector, Classloader, Runtime dataarea, Excution Engine - 가비지 컬렉터는 힙 메모리 영역에 생성된 객체들 중에서 참조되지 않은 객체를 탐색해서 제거한다. - 클래스로더는 클래스 파일을 로딩하여 검증하고 초기화한다. - 런타임 데이터에어리어는 클래스 파일을 저장하고 - 익스큐션 엔진은 클래스 파일을 플랫폼에 맞는 기계어로 변환시켜 실행한다. ### 스레드의 생명주기 스레드 객체가 생성되면 생명주기를 갖게 된다. 크게 5가지로 나눌 수 있다. - New : 스레드가 만들어진 상태인 객체 생성 - Runnable : 실행 대기 상태, 실행 상태로 언제든지 갈 수 있는 상태이다. - Running : 실행 상태, - Blocked : 일시 정지, 사용하고자 하는 객체의 LOCK이 풀릴 때까지 대기 - Waiting : 일시 정지, 다른 스레드가 통지할 때까지 대기 - Timed_waiting : 일시 정지, 주어진 시간 동안 대기 - Terminated dead : 실행을 마친 상태(종료) ### 인터페이스란? 다른 클래스를 작성할 때 기본이 되는 틀을 제공하면서, 다른 클래스 사이의 중간 매개 역할까지 담당하는 일종의 추상 클래스를 의미한다. 추상메소드와 상수만을 포함할 수 있다. ### 다형성이란? 같은 메시지에 대해서 클래스에 따라 다른 행위를 하게 되는 특징이다. 같은 이름을 가지는 메서드에 대해서 인자 개수랑 데이터타입에 따라 수행되는 행위가 달라지는 것을 의미한다. ### 컬렉션 프레임워크란? 다수의 데이터를 쉽고 효과적으로 처리할 수 있는 표준화된 방법을 제공하는 클래스의 집합을 의미하며 즉, 데이터를 저장하는 자료 구조와 데이터를 처리하는 알고리즘을 구조화해서 클래스로 구현해 놓은 것이다. ### Boxing과 UnBoxing? 원시 타입 : int, double, char, byte, short, float, long, boolean 참조 타입 : Integer, Character, Byte, Short, Float, Double, Long, boolean 같은 원시 타입을 참조 타입으로 변환하는 것을 의미한다. 언박싱은 참조 타입을 원시 탕븡로 변환한다. jdk 1.5부터는 오토 박싱, 오토 언박싱 기능이 가능해졌다. ### 스트림의 종류 바이트 입출력의 인풋 스트림, 아웃풋 스트림 문자 입출력의 Reader 와 Writer가 있다. ### jdbc란? 자바에서 db 프로그래밍을 하기 위해 사용되는 api, 즉, 자바 언어와 db를 연결해주는 통로와 같은 것이다. ### PreparedStatement 와 Statement의 차이는 무엇인가? Statement는 정적 쿼리 시 사용하며 매번 파싱과정을 거쳐야 한다. 그래서 부하가 생길 수 있고 sql문을 명확히 알 수 있어 디버깅이 쉽다. PreparedStatement는 동적 쿼리 시 사용하며 한번 파싱하면 그 동일한 sql 문장을 곧바로 파싱 과정 없이 Execution 할 수 있다. 반복적인 다량의 sql 수행 시 성능 상 이득이 있고 오류 발생 시 변수 입력 값을 알 수 없어 디버깅이 어렵다.

Chrome 개발자 도구 사용법

크롬 개발자 도구는 `chrome 크롬 → 설정 → 도구 더보기 → 개발자 도구(F12)` 로 들어갈 수 있다. 개발자 도구를 이용하면 HTML, CSS, JavaScript 의 생산성을 극대화할 수 있다. 자주 쓰는 것들에는 `Elements, Console, Source, Network, Application` 이다. # Elements (요소) - 웹 페이지를 구성하고 있는 요소들을 의미한다. - 왼쪽 상단의 화살표를 클릭하면 요소들을 살펴볼 수 있다. - 오른쪽의 css 창에서 작대기 되있는 것은 우선순위에 밀려있는 것 - Style은 우선순위에 의해 적용되지 않는 것들도 다보여주는 것 - Computed 는 결과적으로 사용자에게 보이는 css를 나열해서 보여주는 것 # aplication (어플리케이션) - 현재 웹페이지가 어떤 리소스로 구성되어있는가로 보여줌 # source (소스) - 웹페이지에 삽입된 자바스크립트, css를 코드 레벨로 볼 수 있다. - 요소는 렌더링된 것을 보여주는 것이고 - 실제로 페이지에 삽입된 코드 자체를 보여준다. # Network (네트워크) - 리소스를 가져오는 과정에서의 데이터 요청에 대한 정보를 가지고 있다. - 엘리멘트 탭에서 보이는 외부 리소스가 네트워크 통신으로부터 받아오는 것들을 의미한다. - FetchX/HR 을 한번 들어가보면 GET/ POST 방식 - GET은 Request URL을 통해서 데이터를 요청하는 방식 - url을 호출해서 url이 서버에 도달하면 원하는 데이터를 반환받는 방식 - POST방식은 URL에 어떤 데이터가 들어가 있지는 않고 - PAYLOAD를 통해서 데이터를 전달하게 되어 있다. # 요약 1. Elements : html/css 상태 확인 2. Console : 콘솔(변수 정보를 확인하고 오류 메세지 표시) 3. Sources : 스크립트 디버깅(break point 지정 및 변수 모니터링 등) 4. Network : 브라우저에서 발생하는 통신 상태 5. Performance(예전 Timeline 패널) : 성능 측정 6. Memory(예전 Profiles 패널) : 메모리 사용 형태를 작성하고 누수 탐색 7. Application(예전 Resources 패널) : 쿠키 및 스토리지 등의 내용 수집 8. Security : Mixed content 이슈, 인증서 문제 등을 디버깅

MSA에 대해서 찾아보기

MSA란 단어를 처음 접한 것은 국비지원교육을 찾아볼 때였다. 한 국비 학원의 교육과정 중 대용량 웹서비스 개발을 위한 MSA Full-Stack 이란 말이 적혀있었다. Okky 커뮤니티에서도 5월 세미나로 실전 MSA 경험 공유라는 제목으로 세미나를 모집했었다. <br /> # MSA? MSA(MicroService Architecture) Martin Folwer는 MSA에 대해 아래와 같이 설명하였다. ``` "the microservice architectural style is an approach to developing a single application as a suite of small services, each running in its own process and communicating with lightweight mechanisms, often an HTTP resource API. These services are built around business capabilities and independently deployable by fully automated deployment machinery." ``` 핵심문구? <br/> > samll services, each running in its own process(스스로 돌아갈 수 있는 작은 서비스) independently deployable(독립적 배포 가능) 하지만 마이크로서비스란 작고, 독립적으로 배포 가능한 각각의 기능을 수행하는 서비스로 구성된 프레임워크 마이크로서비스는 완전히 독립적으로 배포가 가능하고, 다른 기술 스택(개발 언어, DB 등)이 사용 가능한 단일 사업 영역에 초점을 둔다. 독립된 서비스는 API를 통해서만 상호작용한다. ## MSA의 등장배경 기존의 시스템은 정보제공을 위한 화면부터 DB까지 전체가 하나의 흐름으로 묶여 있는 Monolithic Architecture이다. 즉, Monolithic Architecture는 소프트웨어의 모든 구성요소가 한 프로젝트에 통합된 형태<br /> 웹 개발을 예로 들면 웹 프로그램을 개발하기 위해 모듈별로 개발을 하고, 개발이 완료된 웹 어플리케이션을 하나의 결과물로 패키징하여 배포되는 형태를 말한다. 이는 불과 몇년 전까지 대다수의 프로젝트에 적용되었다. Monolithic Architecture의 문제점으로는 1. 서비스/프로젝트가 커지면 커질수록, 영향도 파악 및 전체 시스템 구조의 파악에 어려움 2. 빌드 시간 및 테스트 시간, 배포 시간이 기하급수적으로 늘어나게 됨 3. 서비스를 부분적으로 Scale-out하기가 힘듬 4. 부분의 장애가 전체 서비스의 장애로 이어지는 경우가 발생하게 됨 이러한 상황은 서비스를 수정, 배포하는데 매우 오랜 시간이 걸리고 특정 프레임워크와 언어에 종속적이게 된다. 따라서 MSA가 등장하게 되었다. <br /> ## MSA의 특징 하나의 API를 통해서 모든 구성요소들이 상호작용 한다. 하나의 비즈니스 범위에 맞춰 만들어지기 때문에 하나의 기능만 수행한다. SOA에서 사용되는 집중화된 관리 체계를 사용하지 않으며, REST 등 가벼운 통신 아키텍처 또는 Kafka 등을 이용한 message stream을 주로 사용한다. > `SOA`: Service-Oriented Architecture, 서비스 지향 아키텍처, 서비스 인터페이스를 통한 SW 컴포넌트 재사용 <br /> <br /> ## MSA 장·단점 ### 장점 - 배포(deployment): 서비스 별 개별 배포 가능(배포 시 전체 서비스의 중단이 없음) - 요구사항을 신속하게 반영하여 빠르게 배포할 수 있음 - 확장(scaling) 관점 - 특정 서비스에 대한 확장성이 용이함.<br /> - 클라우드 사용에 적합한 아키텍쳐. - 장애(failure) 관점 - 장애가 전체 서비스로 확장될 가능성이 적음<br /> - 부분적 장애에 대한 격리가 수월함 <br /> ## 단점 - 성능: 서비스 간 호출 시 API를 사용하기 때문에, 통신 비용이나 Latency가 그만큼 증가 - 테스트 / 트랜잭션: 서비스가 분산되어 있어 테스트와 트랜잭션의 복잡도가 증가하고, 많은 자원이 필요 - 데이터: 데이터가 여러 서비스에 걸쳐 분산되기 때문에 한번에 조회하기 어렵고, 데이터의 정합성 또한 관리가 어려움 <br /> ## 번외 : 스프링 클라우드와 MSA 단점에도 불구하고 장점이 훨씬 많은 아키텍처<br /> 스프링 개발 팀은 Spring Framework을 작고 분산되어 클라우드에 쉽게 배포 가능한 서비스를 구축하려는 고도의 분산 모델로 이동하고 있었다. 그로써 스프링 부트와 스프링 클라우드라는 2개의 프로젝트 진행 <br /> 스프링부트는 스프링 프레임워크를 재구성한 프로젝트, 스프링부트의 가벼운 구조는 이후 클라우드 환경에 적용하기 적절한 형태의 프레임워크로 성장함 스프링클라우드는 Public, Private 클라우드에 마이크로서비스를 쉽게 배포하고 운영할 수 있게 돕는다. # 결론 MSA는 작고, 단순하고 분리된 서비스 → 확장 가능, 회복적, 유연한 애플리케이션!<br /> 장단점이 있기에 각각 개발하는 서비스의 백엔드 특성에 맞게 활용 <br /> # 참고 - [알고보니-MSA란?](https://mydream72.tistory.com/entry/%EC%95%8C%EA%B3%A0%EB%B3%B4%EB%8B%88-MSA%EB%9E%80) - [Martinfowler](https://martinfowler.com/articles/microservices.html) - [MSA 제대로 이해하기](https://martinfowler.com/articles/microservices.html) - [MSA로 백엔드 개발을 한다는 것](https://velog.io/@whitebear/MSA) - [알면 좋은 아키텍처, MSA](https://velog.io/@xylopeofficial/알면-좋은-아키텍처-MSA)

교내에서 지도교수님을 필두로 학술동아리를 진행하던 중 이번 학술 활동 주제는 `마이데이터` 관련 주제였다. 생소했던 `마이데이터`라는 단어를 처음 접하게 되었던 계기였고 학술 동아리를 진행하고 나서 곳곳에서 `마이데이터`라는 단어가 점점 눈에 띄기 시작했다. 각자 마이데이터를 이해할 수 있는 자료와 비즈니스 모델 사례, 자기 정보에 관한 디지털 환경, 교육분야의 마이데이터 활용방안, 블록체인 기반 마이데이터 등 마이데이터와 관련된 자료를 읽어보며 내용을 요약정리하여 서로 발표를하며 공유하는 시간을 가졌으며 학술동아리가 마무리될 때 쯤 마침 한국데이터산업진흥원에서 이종분야 개인데이터 융복합 마이데이터 서비스 기획 공모전이 주최되어 학술동아리의 두 팀이 참가해볼 수 있었다. > Github 링크 : [2021-MyData-idea-Competition 2021년 마이데이터 비즈니스 아이디어 공모전](https://github.com/jp0301/2021-MyData-idea-Competition) # DISE(Data In Student Edu) ## 팀 - 팀 명 : 스튜데이터언트 - 팀 원 : 박원석 외 2명 ## 서비스 요약 서비스 명칭 : MyData로 최적화된 My대학 입시 도우미 서비스 분야 : 금융 + 교육, 학습 활용 개인데이터 - 금융 : 개인자산, 보험, 저축, 투자, 신용정보 - 교육 : 학교생활기록부, 성적표, 온/오프라인 수강이력, 학원 정보, 교육 이력 등 교육 프로그램 수강 이력, 학원 정보, 개인 학교 생활 기록부, 성적표 등과 금융 개인 자산 데이터 등을 결합하여 학생 개인 맞춤형 대학 추천, 등록금, 장학금 추천 등 맞춤형 대학 입시 서비스 제공 ## 구축 개요도 <img src="https://raw.githubusercontent.com/jp0301/2021-MyData-idea-Competition/main/구축개요.png" /> ## 서비스 화면 <div className={`flex justify-center h-24`}> <img src="https://raw.githubusercontent.com/jp0301/2021-MyData-idea-Competition/main/UIUX 가이드라인/시작화면.png" /> <img src="https://raw.githubusercontent.com/jp0301/2021-MyData-idea-Competition/main/UIUX 가이드라인/메인화면.png" /> </div> <div className={`flex justify-center h-24`}> <img src="https://raw.githubusercontent.com/jp0301/2021-MyData-idea-Competition/main/UIUX 가이드라인/교육코스화면.png" /> <img src="https://raw.githubusercontent.com/jp0301/2021-MyData-idea-Competition/main/UIUX 가이드라인/교육플랫폼.png" /> </div> <div className={`flex justify-center h-24`}> <img src="https://raw.githubusercontent.com/jp0301/2021-MyData-idea-Competition/main/UIUX 가이드라인/대학진단.png" /> <img src="https://raw.githubusercontent.com/jp0301/2021-MyData-idea-Competition/main/UIUX 가이드라인/마이데이터 정보제공동의화면.png" /> </div> <div className={`flex justify-center h-24`}> <img src="https://raw.githubusercontent.com/jp0301/2021-MyData-idea-Competition/main/UIUX 가이드라인/장학금 분류화면.png" /> </div> ## 결과 열심히 한다고 했지만 되돌아보니 다른 일에 집중하느라 100% 진지하게 임하지 않았던 것 같다. 아쉬움과 좋은 경험을 갖고 `본선진출`까지만 할 수 있었던 경험으로 마무리 하였다. <img src="https://raw.githubusercontent.com/jp0301/2021-MyData-idea-Competition/main/mydata_list.png" />
